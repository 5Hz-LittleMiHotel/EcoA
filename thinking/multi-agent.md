可以这样总结这次工作：

这次我主要是在调试一个本地多 Agent 协作系统。表面上看，问题都是一些小 bug，比如 teammate 状态异常、消息丢失、计划审批卡住、`shutdown` 后又变 `idle`。但真正的难点不是单个 bug，而是：**工程状态机必须顺着大模型的自然行为设计，而不能假设模型会严格按我脑中的协议行动。**

我最后形成的理解是：Agent 工程不是给模型套一堆僵硬规则，而是给它设计“合身的马具”。规则要少，但每条规则都必须承接模型最可能生成的行为。

这次主要克服了几个问题：

1. **消息协议和普通聊天混淆**

最初 teammate 可以用 `send_message` 伪造 `plan_approval_request`。这导致消息看起来像计划请求，但没有生成 `request_id`，也没有进入审批状态机。后来我把普通通信和协议通信拆开：

```text
send_message = 普通聊天
plan_approval = 计划审批协议
shutdown_request / shutdown_response = 关机协议
```

这让我意识到：自然语言消息不能承担状态机职责。状态变化必须走结构化协议。

2. **布尔值无法表达真实意图**

原来的计划审批只有：

```text
approve = true / false
```

但 `false` 有两种含义：

```text
方案不行，修改后重提
任务取消，不要继续
```

模型收到 rejected 后，很自然会尝试修改再提交，因为这符合它的概率生成逻辑。于是我们引入：

```text
next_action = proceed | revise | stop
```

并且默认：

```text
approve=false -> stop
```

只有用户明确要求修改时才 `revise`。这把“计划是否继续”从隐含语义变成用户可见的协议结果。

3. **状态机不能违背用户对“队友”的直觉**

一开始 idle 超时后自动变成 `shutdown`。工程上看似合理，但对大模型来说，teammate 更像一个持续存在的同事，而不是一次性进程。lead 后续很难自然想到要重新 spawn。

所以我把状态语义改成：

```text
idle = 待命，可以被唤醒
shutdown = 显式关闭
```

并让 `send_message` 可以唤醒 idle teammate。这比强迫模型记住“shutdown 后要重启”更合身。

4. **要防止模型陷入协调循环**

当 lead 等待 charlie 提交 plan，但 charlie 一直读文件不提交时，lead 会反复 `read_inbox -> []`，形成协调死循环。解决方式不是骂模型“不要循环”，而是给系统加轻量护栏：

```text
teammate 需要计划时，探索次数有限，之后必须提交 plan
lead 多次读到空 inbox 后，停止轮询并向用户说明等待失败
```

5. **用户必须看见协议结果**

计划被拒绝后，如果用户不知道 teammate 是停止、重提还是继续等待，就会误以为系统卡住。因此我加入了 Plan Review Receipt，主动展示：

```text
谁提交了计划
计划内容是什么
审批结果是什么
为什么
下一步是 proceed / revise / stop
```

这让 Agent 的内部状态对用户可解释。

可以对面试官这样提炼：

> 我这次最大的收获是，Agent 工程的核心不是把传统程序状态机硬套到大模型上，而是观察模型自然会怎么补全、怎么误解、怎么尝试继续完成任务，然后把这些高概率行为纳入协议设计。好的 Agent 框架应该少规则，但规则要合身：普通消息和协议消息分离，状态转移结构化，拒绝后的下一步显式化，idle 和 shutdown 的语义符合用户对队友的直觉。  
>   
> 这和 Claude Code 的设计哲学是一致的：不是用复杂控制压制模型，而是给模型刚好够用的工具、反馈和边界，让它自然地做对事。
